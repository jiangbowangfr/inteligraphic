# Summary and Delivery Guide

> **Purpose:** Describe the structure, build, test and execution of the project:
> - What exactly was done to produce the `DFT-Studio-Project/Project/` folder?
> - What is tested, and what does each test verify?
> - How does the release team build, run tests, and use the binary?

---

## 1. What Was Built (and Why)

The following requirement is set out:

> *DFT Studio needs to read LEF+DEF files, extract die area / master dimensions / instance placements / regions into an in-house data structure, and expose that data to the GUI. OpenDB (from OpenROAD) is used as the parsing back-end. The integration must be deliverable as a self-contained CMake sub-project — no Tcl, no SWIG, no other OpenROAD tools.*

The result is `DFT-Studio-Project/Project/src/`, a fully independent C++ project that wraps the necessary OpenDB subsystems behind a clean, narrow API.

## 2. File Provenance — What Was Copied, Generated, and Written

The entire deliverable lives under `Project/src/`. Every file falls into one of four categories.

### 2.1 Copied verbatim from OpenROAD `src/`

These files were taken from the OpenROAD monorepo (under `src/odb/` and `src/utl/`) with **no content changes**. They are placed under `vendor/` to make the sub-project self-contained.

| Destination in `vendor/` | Upstream source in OpenROAD |
|---|---|
| `vendor/odb/include/odb/` | `src/odb/include/odb/` (all public headers) |
| `vendor/odb/src/db/` | `src/odb/src/db/` (OpenDB core object model) |
| `vendor/odb/src/lef/` | `src/odb/src/lef/` (LEF grammar + scanner, minus generated outputs) |
| `vendor/odb/src/lefin/` | `src/odb/src/lefin/` (LEF → OpenDB importer) |
| `vendor/odb/src/lefout/` | `src/odb/src/lefout/` (LEF writer; kept because it is transitively referenced by db internals) |
| `vendor/odb/src/def/` | `src/odb/src/def/` (DEF grammar + scanner, minus generated outputs) |
| `vendor/odb/src/defin/` | `src/odb/src/defin/` (DEF → OpenDB importer) |
| `vendor/odb/src/defout/` | `src/odb/src/defout/` (DEF writer; same rationale as lefout) |
| `vendor/odb/src/zutil/` | `src/odb/src/zutil/` (geometry utilities) |
| `vendor/odb/src/cdl/` | `src/odb/src/cdl/` (circuit description layer; required by db) |
| `vendor/utl/include/utl/` | `src/utl/include/utl/` (logger / timer headers) |
| `vendor/utl/src/` (most files) | `src/utl/src/` (logger / timer / metrics implementation) |

**Total vendored files:** ~330 headers + ~362 `.cpp` source files.

### 2.2 Generated by Bison, then committed

The LEF and DEF parsers are produced by Bison from `.y` grammar files. Rather than requiring Bison at build time on every developer machine, the generated C++ files are committed alongside the grammar sources.

| Generated file | Grammar source | Generator |
|---|---|---|
| `vendor/odb/src/lef/generated/lef_parser.cpp` | `vendor/odb/src/lef/lef/lef.y` | `bison --output=...` |
| `vendor/odb/src/lef/generated/lef_parser.hpp` | same | same |
| `vendor/odb/src/def/generated/def_parser.cpp` | `vendor/odb/src/def/def/def.y` | `bison --output=...` |
| `vendor/odb/src/def/generated/def_parser.hpp` | same | same |

The regeneration script is preserved at `vendor/tools/regen_parsers.sh`. It must only be re-run when syncing upstream grammar file changes:

```bash
cd Project/src/vendor
bash tools/regen_parsers.sh   # requires bison >= 3.0
# then commit the newly generated files
```

### 2.3 Copied and then modified

These two files came from the OpenROAD source tree but required targeted changes to remove Tcl and Prometheus dependencies that are not available in the standalone build.

| File | Upstream source | Changes made |
|---|---|---|
| `vendor/utl/src/decode.cpp` | `src/utl/src/decode.cpp` | Wrapped `#include "tcl.h"` and `evalTclInit()` in `#ifndef DFTODB_NO_TCL` guards |
| `vendor/utl/include/utl/decode.h` | `src/utl/include/utl/decode.h` | Same `DFTODB_NO_TCL` guard on the `evalTclInit` declaration |

The exact diff is recorded in `vendor/patches/utl_decode_no_tcl.patch` for traceability. The compile-time flags `DFTODB_NO_TCL` and `DFTODB_NO_PROMETHEUS` are set on the `utl_lib` CMake target in `vendor/CMakeLists.txt`.

### 2.4 Written from scratch (all new code)

Everything outside `vendor/` is new code written for this project.

#### Public API layer — `dft_api/`

## 3. Unit Tests Covered

There are two GTest executables. Both are built unconditionally when `DFTODB_BUILD_TESTS=ON` and run via `ctest`.

### 3.1 `test_import_and_traverse` (14 test cases)

All cases use the two-macro (`B1` / `B2`) fixture under `tests/data/`.

| Test case | What it verifies |
|---|---|
| `HandleIsValid` | `LoadLefDef` returns a valid `DesignHandle` on well-formed input |
| `DbuPerMicron` | `GetDbuPerMicron()` returns the value declared in `UNITS DISTANCE MICRONS` (1000) |
| `DieArea` | `GetDieArea()` returns the exact rectangle from `DIEAREA ( 0 0 ) ( 1000 1000 )` |
| `MasterCount` | `ForEachMaster` visits exactly 2 masters |
| `MasterOrderAndSize` | Masters are iterated alphabetically (`B1` before `B2`); exact DBU dimensions checked |
| `MasterEarlyExit` | Returning `false` from the callback stops iteration immediately |
| `InstanceCount` | `ForEachInstance` visits exactly 2 instances |
| `InstanceNamesAndMasters` | Each instance references the correct master by name |
| `InstanceOrigins` | Placed origins `(100,200)` and `(400,100)` match the DEF |
| `InstanceBoundingBoxes` | `compute_bbox=true` produces correct `ll/ur` by adding master size to origin |
| `InstanceOrientations` | Both instances report `Orient::N` |
| `InstanceEarlyExit` | Early-exit behaviour on instance iteration |
| `TraverserFromConstHandle` | `Traverser` can be constructed from a `const DesignHandle&` |

### 3.2 `test_negative_inputs` (6 test cases)

| Test case | What it verifies |
|---|---|
| `MissingLefFile` | Non-existent LEF path → `status.ok == false`, message non-empty |
| `MissingDefFile` | Non-existent DEF path → `status.ok == false` |
| `EmptyLefFileList` | Passing zero LEF files → `status.ok == false` |
| `MalformedLef` | LEF file containing only garbage text → `status.ok == false`, no crash |
| `EmptyDef` | DEF file containing only whitespace → `status.ok == false`, no crash |
| `DefaultConstructedHandleIsInvalid` | `DesignHandle{}` constructed without loading → `is_valid() == false` |

### 4.4 Known gaps (TC-05 and TC-07)

### 4.4 已知限制

**TC-05 — BLOCKAGES（已解决）：** TC-05 的 `BLOCKAGES` 小节曾触发 ODB-0421 导致整个 DEF 文件解析失败。已在 `Importer::LoadLefDef()` 中添加 DEF 预处理：在交给 OpenDB 读取器之前，自动剥除所有 `BLOCKAGES … END BLOCKAGES` 块。TC-05 现在 Import OK，die/masters/instances/regions 均正确输出。BLOCKAGES 几何数据本身不包含在 JSON 输出中，规划 v2 补全。

**TC-07 — 多边形 REGIONS + 前向引用（已知，规划中）：** TC-07 有两个相互叠加的问题：
1. `COMPONENTS` 小节先于 `REGIONS` 出现，导致 OpenDB 在处理实例时创建空 region 对象，之后 `REGIONS` 小节再次创建同名 region，触发 ODB-0152 "Region already exists" 警告。
2. `REGIONS` 使用多边形点列语法（4 个角点）而非标准的两点矩形语法，触发 ODB-0421 解析错误。

这两个问题超出了 BLOCKAGES 剥离方案的修复范围，暂时归类为 v2 待办事项。TC-07 当前仍以 exit 1 退出，并在 `actual-produced.json` 中输出 `{"error": "..."}`。

**PINS 未提取（所有 TC）：** DEF `PINS` 小节由 OpenDB 正常读取解析，但 `Traverser` 目前没有 `ForEachTopPin` 方法，`actual-produced.json` 不包含 `pins` 字段。规划 v2 新增该 API。

## 5. Batch Test Script

`Project/src/run_tc_tests.sh` runs `dftodb-inspect` against every scenario directory.

### 5.1 Usage

```bash
cd Project/src           # must be run from here (binary is at ./build/tools/)

./run_tc_tests.sh                    # run all 10 scenarios; human-readable tables
./run_tc_tests.sh 04 09              # run TC-04 and TC-09 only
./run_tc_tests.sh --json             # run all scenarios; print JSON output
./run_tc_tests.sh --compact 01       # run TC-01 with compact (single-line) JSON
./run_tc_tests.sh --json 04 09       # run TC-04 and TC-09 with JSON output
```

The script exits with code `0` if all executed scenarios pass, non-zero otherwise — suitable for CI.

### 5.2 How the script detects test cases

For each scenario directory (sorted numerically) it:
1. Adds `tech.lef` and `lib.lef` (or `libA.lef` + `libB.lef`, etc.) in discovery order.
2. Runs `dftodb-inspect [--lef ...] --def design.def [--text|--json|--compact]`.
3. Records PASS if the binary exits with code 0, FAIL otherwise.

Note: The script currently tests only that the binary loads and traverses without crashing (exit code). For a full field-level semantic comparison against `expected.json`, use the dedicated Python script:

```bash
# Generate (or refresh) actual-produced.json for all TCs, then compare:
bash gen_actual_json.sh 2>/dev/null
python3 compare_expected_vs_actual.py
```

`compare_expected_vs_actual.py` prints a per-TC pass/fail table and exits non-zero if any field mismatches are found. Manual comparison against `expected.json` can also be done with `--json` mode and a JSON diff tool:

```bash
./build/tools/dftodb-inspect \
  --lef ../doc/def-tests/dft_opendb_testcases/01_basic_two_blocks/tech.lef \
  --lef ../doc/def-tests/dft_opendb_testcases/01_basic_two_blocks/lib.lef \
  --def ../doc/def-tests/dft_opendb_testcases/01_basic_two_blocks/design.def \
  2>/dev/null > /tmp/actual.json

diff <(jq -S . /tmp/actual.json) \
     <(jq -S . ../doc/def-tests/dft_opendb_testcases/01_basic_two_blocks/expected.json)
```

## 7. Using the `dftodb-inspect` Binary

`dftodb-inspect` is both a functional CLI tool and a demonstration of every public API call. Its source (`tools/dftodb_inspect.cpp`) can be read as a worked example.

### 7.1 Synopsis

```
dftodb-inspect --lef <lef> [--lef <lef> ...] --def <def> [OPTIONS]
```

At least one `--lef` and exactly one `--def` are required. LEF files are processed in the order given; the technology LEF (`tech.lef`) must come first.

### 7.2 Output format options

| Flag | Behaviour |
|---|---|
| *(none)* | JSON output, pretty-printed (default) |
| `--json` | Explicit pretty JSON |
| `--compact` | JSON on a single line (no whitespace) |
| `--text` | Human-readable ASCII table |

### 7.3 JSON output schema

```
{
  "dbu_per_micron": <int>,        // coordinate units: divide by this to get µm

  "die": {                         // DEF DIEAREA
    "xMin": <int>, "yMin": <int>, "xMax": <int>, "yMax": <int>
  },

  "masters": [                     // one entry per LEF MACRO
    {
      "name": <string>,
      "bbox": { "xMin": <int>, "yMin": <int>, "xMax": <int>, "yMax": <int> }
                                   // origin-relative; width = xMax-xMin
    }
  ],

  "instances": [                   // one entry per DEF COMPONENT
    {
      "name":      <string>,       // instance name
      "master":    <string>,       // LEF macro name
      "origin":    { "x": <int>, "y": <int> },   // placed lower-left in DBU
      "orient":    <string>,       // "N"|"S"|"E"|"W"|"FN"|"FS"|"FE"|"FW"
      "placement": <string>,       // "PLACED"|"FIXED"|"COVER"|"FIRM"|"UNPLACED"
      "region":    <string>|null,  // DEF REGION name, or null
      "bbox":      { "xMin": <int>, "yMin": <int>, "xMax": <int>, "yMax": <int> }
                                   // placed bbox in top-level DBU coordinates
    }
  ],

  "regions": [                     // one entry per DEF REGIONS entry
    {
      "name":  <string>,
      "rects": [ { "xMin": <int>, ... }, ... ]
    }
  ]
}
```

### 7.4 Text mode output (`--text`)

```
=== Design ===
  DBU per µm : 1000
  Die area   : (0, 0) -> (2000, 2000) DBU

=== Masters (LEF macros) ===
Name                            Width (DBU)   Height (DBU)
B1                              100000        80000
B2                              40000         50000

=== Instances (DEF components) ===
Instance   Master  Status   Orient  Origin (DBU)   Region  BBox LL->UR (DBU)
u_b1       B1      PLACED   N       (100, 200)     -       (100,200)->(100100,80200)
u_b2       B2      PLACED   N       (400, 100)     -       (400,100)->(40400,50100)

=== Regions (DEF REGIONS) ===
  (none)
```

Add `--um` to print all coordinates in µm instead of DBU.

### 7.5 Selective section output (text mode only)

```bash
dftodb-inspect --lef tech.lef --lef lib.lef --def design.def \
    --text --show-instances          # print only the instances table
```

Flags: `--show-design`, `--show-masters`, `--show-instances`, `--show-regions`.  
If none are given, all sections are printed.

### 7.6 DEF parser options

By default, net wires, pin-to-net connections, and power routing are skipped (fastest import). To enable them:

```bash
dftodb-inspect ... --parse-wires \
                   --parse-connections \
                   --parse-special-wires
```

`--continue-on-errors` causes non-fatal DEF warnings to be printed without aborting the load.

### 7.7 Practical examples

```bash
# Default JSON output — pipe to jq for formatting/querying
./build/tools/dftodb-inspect \
    --lef path/to/tech.lef \
    --lef path/to/macros.lef \
    --def path/to/design.def 2>/dev/null | jq '.instances | length'

# Compact JSON — suitable for piping to a downstream tool
./build/tools/dftodb-inspect \
    --lef tech.lef --lef lib.lef --def design.def \
    --compact 2>/dev/null

# Human-readable table output
./build/tools/dftodb-inspect \
    --lef tech.lef --lef lib.lef --def design.def \
    --text 2>/dev/null

# µm units in text mode
./build/tools/dftodb-inspect \
    --lef tech.lef --lef lib.lef --def design.def \
    --text --um 2>/dev/null

# Run all batch test scenarios (text mode)
cd Project/src
./run_tc_tests.sh

# Run a single scenario in JSON mode
./run_tc_tests.sh --json 04
```

## 8. API Quick-Start (C++ Integration)

```cpp
#include "dftodb/Importer.h"
#include "dftodb/Traverser.h"
#include "dftodb/Json.h"

// 1. Load
auto [status, handle] = dftodb::Importer::LoadLefDef(
    {"tech.lef", "lib.lef"},   // LEF files, tech first
    "design.def",              // DEF file
    {}                         // default ImportOptions
);
if (!status.ok) {
    std::cerr << status.message << "\n";
    return 1;
}

// 2. Traverse
dftodb::Traverser t(handle);

t.ForEachMaster([](const dftodb::MasterInfo& m) {
    // m.name, m.bbox.width(), m.bbox.height()
    return true;  // continue
});

t.ForEachInstance([](const dftodb::InstanceInfo& inst) {
    // inst.name, inst.master_name, inst.origin, inst.orient,
    // inst.placement, inst.region_name, inst.bbox
    return true;
}, /* compute_bbox= */ true);

t.ForEachRegion([](const dftodb::RegionInfo& r) {
    // r.name, r.rects
    return true;
});

// 3. Serialize to JSON
std::string json = dftodb::ToJson(t);           // pretty-printed
std::string compact = dftodb::ToJson(t, false); // compact
```

CMake integration (target name):

```cmake
target_link_libraries(my_target PRIVATE dftodb::dft_api)
```
